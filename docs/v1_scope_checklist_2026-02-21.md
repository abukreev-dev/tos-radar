# V1 Scope And Implementation Checklist

Дата: 2026-02-21  
Статус: готово к реализации  
Источник решений: `docs/client_ui_context_2026-02-21.md` (Q/A closed)

## 1) Scope v1 (что делаем)

- Приватный кабинет: `Дашборд`, `Каталог`, `Мои сервисы`, `Изменения`, `Тариф/Биллинг`, `Профиль`, `Поддержка`.
- Подключение сервисов только из проверенного каталога (добавление URL пользователем отсутствует).
- Проверки ToS по расписанию 1 раз в сутки.
- Уведомления: Email + Telegram, формат `ежедневный дайджест`.
- Интеграции в v1: только Telegram (через `Профиль -> Уведомления`).
- Биллинг v1: тестовый баланс, тарифные ограничения, переходы Free/Paid по утвержденным правилам.
- Аккаунт: email+password, Google login, verify email flow, удаление аккаунта (soft delete 30 дней).
- Стандартизованные состояния ошибок UI: `403`, `404`, `500`.

## 2) Что явно не входит в v1

- Новые продуктовые вопросы и расширения без отдельного решения.
- Дополнительные интеграции (`Slack`, `Webhook`, `Zapier`).
- Глобальный поиск по кабинету.
- Экспорты из раздела `Изменения`.
- Bulk-операции в `Мои сервисы`.
- In-app notification center (колокольчик).
- API/Webhooks для клиентов.
- Реферальная механика.

Правило по умолчанию: все новые спорные UI-пункты = `deferred post-v1`.

## 3) Реализационный чеклист (MVP-ready)

## 3.1 Frontend

- Реализовать навигацию: desktop sidebar + mobile bottom tab bar (как зафиксировано).
- Дашборд:
- KPI за 24ч + блок `Последние изменения`.
- CTA `Включить уведомления` при выключенных каналах.
- Компактный onboarding-чеклист (скрываемый).
- Каталог:
- Поиск по названию/домену.
- Подключение сервиса по кнопке `Подключить`.
- Paywall-паттерн для ограничений тарифа.
- Мои сервисы:
- Поиск по названию/домену.
- Порядок по умолчанию: по дате подключения.
- В карточке: дата подключения, `Открыть источник`, `Последнее изменение` (если есть).
- Удаление через confirm-модалку.
- Изменения:
- Карточки + пагинация.
- Фильтры: период + сервис.
- Без фильтра `change_level` в v1.
- Empty-state с CTA `Сбросить фильтры`.
- Профиль -> Уведомления:
- Управление email/telegram дайджестом.
- Кнопка `Отправить тест` для Telegram с лимитами.
- Кнопка resend verify email с лимитами.
- Индикаторы статусов каналов и ошибок доставки.
- `Сохранить` (без autosave).
- Страницы ошибок: `403`, `404`, `500` (для `500` показывать короткий ID ошибки).

## 3.2 Backend

- Проверка доступности каналов и ограничений:
- Email-дайджест только при verify email.
- Telegram-канал с flow привязки через bot + code.
- Автосброс Telegram-тумблеров при потере связки.
- Логика дайджеста:
- Ежедневная отправка в 09:00 timezone профиля.
- Fallback timezone: `Europe/Moscow` + маркер в UI.
- Нет ручной отправки `Send now`.
- Нет retry по кнопке `Повторить позже` (только скрытие alert до следующего окна).
- Ошибки доставки:
- Hard-bounce email => выключение email-канала + причина в UI.
- Смена email => сброс email-дайджеста до повторного verify.
- Аккаунт и безопасность:
- Logout всех сессий при смене пароля.
- Soft delete 30 дней + восстановление в 1 клик.
- Аудит/трекинг:
- Не добавлять события для `закрыл onboarding` и `включил первый канал`.
- Не добавлять UTM/source tracking переходов в биллинг.

## 3.3 QA/Acceptance

- Проверить тарифные ограничения и paywall-потоки (Free/Paid).
- Проверить сценарии verify email до/после включения каналов.
- Проверить Telegram flow: link, unlink, disconnected, test-send limits.
- Проверить все error states (`403`, `404`, `500`) и тексты.
- Проверить `Профиль -> Уведомления` на mobile/desktop.
- Проверить soft-delete lifecycle (вход, восстановление, финализация).
- Проверить отказоустойчивость сохранения настроек (`Сохранить`, retry, блок повторных кликов).

## 4) Definition of Done для старта релиза

- Все пункты из разделов `3.1` и `3.2` реализованы.
- Чеклист `3.3` пройден на staging.
- Открытых блокеров `P0/P1` нет.
- Любые новые идеи после этой фиксации документируются как `post-v1`.
